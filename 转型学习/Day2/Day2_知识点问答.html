1 位置参数和关键字参数有什么区别？* 在函数定义中怎么用？ ▼

函数进阶

位置参数（positional arguments）：按调用时传入的位置顺序匹配参数名。

关键字参数（keyword arguments）：用 name=value 形式指定参数名传参，顺序可以任意。

def greet(name, greeting, punctuation="!"):
    print(f"{greeting}, {name}{punctuation}")

# 位置参数
greet("小明", "你好")           # → 你好, 小明!

# 关键字参数
greet(greeting="Hello", name="Alice")  # 顺序随便

# 混合：位置参数必须在关键字参数之前
greet("Bob", greeting="Hi")

单独的 *（在参数列表中）：表示后面的参数只能用关键字传参，强制明确。

def f(a, b, *, c, d):
    pass

f(1, 2, c=3, d=4)   # ✅ 正确
f(1, 2, 3, 4)       # ❌ TypeError: c 是仅限关键字参数

为什么有用？LangChain/FastAPI 大量使用这种模式，让 API 调用更可读、不易出错。

2 *args 是什么？怎么用？ ▼

函数进阶

*args 用来接收任意数量的位置参数，在函数内部是一个元组（tuple）。

def sum_all(*args):
    print(type(args))  # 
    return sum(args)

print(sum_all(1, 2, 3, 4))  # → 10
print(sum_all())             # → 0

也可以配合普通参数：

def log(level, *messages):
    for msg in messages:
        print(f"[{level}] {msg}")

log("INFO", "启动", "连接成功", "处理完成")

拆包（unpacking）：在调用时用 * 把列表/元组展开：

nums = [10, 20, 30]
print(sum_all(*nums))  # 等价于 sum_all(10, 20, 30)

类比 Swift：可变参数 func sum(_ numbers: Int...) -> Int。

3 **kwargs 是什么？和 *args 有什么区别？ ▼

函数进阶

**kwargs 用来接收任意数量的关键字参数，在函数内部是一个字典（dict）。

def print_config(**kwargs):
    for key, value in kwargs.items():
        print(f"{key} = {value}")

print_config(host="localhost", port=8080, debug=True)
# 输出:
# host = localhost
# port = 8080
# debug = True

与 *args 的区别：

 *args**kwargs
接收位置参数关键字参数
类型tupledict
调用解包func(*list)func(**dict)

拆包调用：

config = {"host": "localhost", "port": 8080}
print_config(**config)  # 把字典展开成关键字参数

为什么 AI 库到处都是 **kwargs？因为 API 函数需要透传大量参数给底层，用 **kwargs 就不用把所有参数都显式写出来。例如 ChatOpenAI(model="...", temperature=0.7, **kwargs)。

4 默认参数陷阱是什么？def f(x=[]) 为什么是坑？ ▼

函数进阶

Python 的默认参数值只在函数定义时创建一次，之后每次调用都复用同一个对象。

def add_item(item, items=[]):
    items.append(item)
    return items

print(add_item("a"))  # → ['a']
print(add_item("b"))  # → ['a', 'b']   ← 不是 ['b']！
print(add_item("c"))  # → ['a', 'b', 'c']

原因：默认列表 [] 在函数定义时只创建一次，每次调用共享同一个列表。

正确做法：用 None 作为默认值，在函数体内创建：

def add_item(item, items=None):
    if items is None:
        items = []
    items.append(item)
    return items

同样的坑也适用于：def f(d={})（可变字典）、def f(s=set())（可变集合）。不可变类型（int, str, tuple）不会有这个问题。

5 Lambda 函数是什么？和 Swift 闭包有什么对应关系？ ▼

函数进阶

Lambda 是 Python 的匿名函数，一行写完，自动 return：

# 语法：lambda 参数: 表达式
double = lambda x: x * 2
print(double(5))  # → 10

# 常用于排序、map/filter
nums = [3, 1, 4, 1, 5]
nums.sort(key=lambda x: -x)  # 按降序排列

# map: 对每个元素应用函数
list(map(lambda x: x**2, [1, 2, 3]))  # → [1, 4, 9]

# filter: 过滤
list(filter(lambda x: x > 2, [1, 2, 3, 4]))  # → [3, 4]

Swift 对照：

// Swift
let double = { $0 * 2 }
double(5)  // → 10

// Python
double = lambda x: x * 2
double(5)  # → 10

限制：Lambda 只能写一个表达式，不能写语句（不能有 if/for/print 等）。复杂逻辑用普通 def。

6 闭包（Closure）是什么？怎么用 nonlocal？ ▼

函数进阶

闭包：一个内部函数"记住"并访问了外部函数的变量，即使外部函数已经执行完毕。

def make_counter():
    count = 0        # 外部函数的局部变量
    def counter():   # 内部函数（闭包）
        nonlocal count  # 声明要修改外部变量
        count += 1
        return count
    return counter   # 返回内部函数

# 使用
c1 = make_counter()
print(c1())  # → 1
print(c1())  # → 2
print(c1())  # → 3

# 每个闭包独立
c2 = make_counter()
print(c2())  # → 1 (新的独立计数器)

nonlocal 的作用：告诉 Python "我要修改外层函数的变量"，不加的话会创建一个新的局部变量。

和 Swift 的对比：Swift 闭包也有捕获（capture）机制，默认捕获是强引用，需要用 [weak self] 避免循环引用。

实际应用：装饰器（decorator）本质上就是用闭包实现的——装饰器是一个函数 A，它接收一个函数 B，返回一个闭包 C，C 包装了 B 并添加了额外逻辑。

7 装饰器（Decorator）的原理是什么？和闭包的关系？ ▼

函数进阶

装饰器本质就是一个接收函数、返回新函数的函数（底层就是闭包）。

def timer(func):          # 装饰器函数
    def wrapper(*args, **kwargs):  # 闭包
        import time
        start = time.time()
        result = func(*args, **kwargs)
        print(f"耗时: {time.time()-start:.3f}s")
        return result
    return wrapper

@timer                    # 语法糖：等价于 slow_func = timer(slow_func)
def slow_func():
    import time
    time.sleep(1)
    return "done"

slow_func()  # 输出: 耗时: 1.003s

不加 @ 的等价写法：

def slow_func():
    ...
slow_func = timer(slow_func)  # 手动装饰

装饰器和闭包的对比：

  • 闭包是"内部函数引用外部变量"的机制
  • 装饰器是"用闭包包装另一个函数"的模式
  • 装饰器是闭包的典型应用，但不是闭包的全部

FastAPI、Flask、pytest 大量使用装饰器。比如 @app.get("/")、@pytest.fixture。

9 self 在 Python 里为什么必须写？ ▼

面向对象

这是 Python 和 Swift 最明显的差异之一。

Python 的设计哲学："显式优于隐式"（Explicit is better than implicit）。

class Dog:
    def __init__(self, name):  # self 就是"这个实例自己"
        self.name = name       # self.name 是实例属性
    
    def bark(self):            # 每个方法第一个参数都是 self
        print(f"{self.name} 汪汪!")

# 调用时，Python 会自动把实例传进去
d = Dog("旺财")
d.bark()  # 等价于 Dog.bark(d)

# 所以 self 就是实例本身，相当于 Swift 的 self

Swift 对比：

// Swift 里 self 是隐式的，不写也行
class Dog {
    var name: String
    func bark() { print("\(name) 汪汪!") }
}

# Python 必须写 self，不写会报错
class Dog:
    def bark(self):  # 少写 self 会报: takes 0 positional arguments but 1 was given
        print("汪汪!")

为什么 Python 要这样设计？因为 Python 的方法本质就是函数，instance.method() ≈ Class.method(instance)，所以必须明确把实例传进来。

10 @property 是什么？等于 Swift 的什么？ ▼

面向对象

@property 把方法变成像属性一样访问，相当于 Swift 的计算属性（computed property）。

class Circle:
    def __init__(self, radius):
        self.radius = radius

    @property
    def area(self):       # 像属性一样调用，不用加 ()
        return 3.14 * self.radius ** 2

    @property
    def diameter(self):
        return self.radius * 2

c = Circle(5)
print(c.area)      # 不用 c.area()
print(c.diameter)  # 不用 c.diameter()

# 也可以设置 setter（类似 Swift 的 set）
@area.setter
def area(self, value):
    self.radius = (value / 3.14) ** 0.5

Swift 对照：

// Swift
class Circle {
    var radius: Double
    var area: Double {  // 计算属性
        return 3.14 * radius * radius
    }
}

为什么有用？在 LLMClient 中 history_length 就是用 @property 实现的，调用方用 client.history_length 而不是 client.history_length()，代码更简洁。

11 @dataclass 是什么？它自动帮我们生成了什么？ ▼

面向对象

@dataclass 是 Python 3.7+ 引入的装饰器，自动生成以下方法，省去大量重复代码：

from dataclasses import dataclass

@dataclass
class ChatMessage:
    role: str
    content: str

# ⬆ 相当于手动写了下面全部 ⬇

class ChatMessage:
    def __init__(self, role: str, content: str):
        self.role = role
        self.content = content
    def __repr__(self):          # 打印对象
        return f"ChatMessage(role='{self.role}', content='{self.content}')"
    def __eq__(self, other):     # 比较两个对象
        return self.role == other.role and self.content == other.content

自动生成的方法：

  • __init__ — 构造函数
  • __repr__ — 调试打印（print(msg) 显示内容）
  • __eq__ — 比较相等（msg1 == msg2）
  • 可选：__hash__、__lt__ 等

和 Swift 的对比：@dataclass ≈ Swift 的 struct（自动生成初始化器和 Equatable 等）。

12 field(default_factory=...) 是什么？为什么需要它？ ▼

面向对象

这是 dataclass 中处理可变默认值的方式。

问题：dataclass 也不允许直接用可变对象做默认值（同样有"默认参数陷阱"）：

@dataclass
class Client:
    history: list = []  # ❌ 会报错：ValueError
    # 原因：所有实例会共享同一个 list

解决方案：用 field(default_factory=...)，每次创建实例时调用工厂函数生成新对象：

from dataclasses import dataclass, field

@dataclass
class LLMClient:
    api_key: str = field(default_factory=lambda: os.getenv("OPENAI_API_KEY"))
    # 等价于：每次创建实例都调用一次 os.getenv("OPENAI_API_KEY")

    _history: list = field(default_factory=list)
    # 每次创建都新建一个空列表，互不共享

lambda 在这里的作用：不是直接赋值，而是提供一个"创建函数"，每次需要默认值时调用它。

常见用途：从环境变量读配置、创建空列表/字典/集合。

13 __post_init__ 是什么？什么时候会被调用？ ▼

面向对象

__post_init__ 是 dataclass 提供的一个钩子方法，在 __init__ 执行完毕后自动调用。

@dataclass
class LLMClient:
    api_key: str = ...  # dataclass 自动生成 __init__

    def __post_init__(self):
        # ⭐ 这里的代码会在 __init__ 完成后自动执行
        self._client = OpenAI(api_key=self.api_key, base_url=self.base_url)
        self._history = []
        self.reset()

# 创建实例时：
client = LLMClient()
# 1️⃣ dataclass 自动执行生成的 __init__（给字段赋值）
# 2️⃣ 自动调用 __post_init__(self)
# 3️⃣ 所以在 __post_init__ 里可以安全地使用 self.api_key

为什么需要它？

  • dataclass 的 __init__ 只能做字段赋值
  • 如果你还需要做额外初始化（创建其他对象、调用方法），就在 __post_init__ 里做
  • 相当于 Swift 中在 init 里写额外逻辑

15 __str__ 和 __repr__ 是什么？有什么区别？ ▼

面向对象

这两个方法控制对象"怎么被打印"。

class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y
    
    def __repr__(self):   # 给开发者看，要包含足够信息
        return f"Point(x={self.x}, y={self.y})"
    
    def __str__(self):    # 给用户看，要简洁美观
        return f"({self.x}, {self.y})"

p = Point(3, 4)
print(repr(p))  # → Point(x=3, y=4)
print(str(p))   # → (3, 4)
print(p)        # 默认调用 __str__

区别总结：

 __repr____str__
目标用户开发者最终用户
调用时机repr(obj)、调试器print(obj)、str(obj)
如果没有显示 <__main__.Point object at 0x...>退回到 __repr__
最佳实践始终实现 __repr__需要时再实现

Swift 对照：__repr__ ≈ CustomDebugStringConvertible，__str__ ≈ CustomStringConvertible。

16 Python 如何实现继承？子类怎么调用父类方法？ ▼

面向对象

继承语法：

class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        return "..."

class Dog(Animal):          # 括号里写父类
    def speak(self):        # 重写（override）
        return f"{self.name}: 汪汪!"

class Cat(Animal):
    def speak(self):
        return f"{self.name}: 喵喵!"

d = Dog("旺财")
print(d.speak())  # → 旺财: 汪汪!

子类调用父类方法：用 super()

class Puppy(Dog):
    def __init__(self, name, cuteness=10):
        super().__init__(name)   # 调用父类的 __init__
        self.cuteness = cuteness

    def speak(self):
        parent_sound = super().speak()  # 调用父类的 speak
        return f"{parent_sound} (可爱度: {self.cuteness})"

和 Swift 的对比：

// Swift
class Dog: Animal {
    override func speak() -> String {
        return "汪汪!"
    }
}
// 调用父类：super.speak()

Python 用 super()，Swift 用 super，概念完全一样。

18 LLMClient 的 _history 是干什么的？对话历史是怎么维护的？ ▼

LLMClient

_history 是一个 List[ChatMessage]，按顺序存储所有消息：

    ChatMessage(role="system", content="你是一个有帮助的AI助手。")

20 CLI 中的 if __name__ == "__main__": 是什么？ ▼

LLMClient

if __name__ == "__main__":
    main()

这行代码的作用是：只有直接运行这个文件时才执行 main()，如果被别的文件 import 则不执行。

21 生成器（Generator）和 yield 是怎么工作的？ ▼

LLMClient

生成器是一个"可以暂停和恢复"的函数。每次遇到 yield，函数会"冻结"当前状态，返回一个值，下次调用时从暂停处继续。

def count_up_to(n):
    i = 0
    while i < n:
        yield i      # 暂停，返回 i
        i += 1       # 下次继续从这里执行

# 使用
for num in count_up_to(3):
    print(num)
# 输出: 0, 1, 2

stream_chat 中的实际应用：

def stream_chat(self, user_input: str) -> Generator[str, None, None]:
    # ... 调用 API 时设置 stream=True ...
    full_reply = ""
    for chunk in stream:         # API 逐个返回 chunk
        delta = chunk.choices[0].delta.content
        if delta:
            full_reply += delta
            yield delta           # 每收到一个字就返回给调用方
    # 所有 chunk 处理完后，把完整回复加入历史
    self._history.append(...)

和普通函数的区别：

 returnyield
行为函数结束，释放所有状态函数暂停，保留状态

24 Mock 是什么？为什么测试 LLMClient 需要它？ ▼

pytest

Mock（模拟对象）是一个"假的对象"，可以代替真实对象，让你控制它的行为。

为什么需要 Mock？

# 测试 chat() 时如果不 mock：
client = LLMClient()
client.chat("你好")
# → 真的去调 API 了！
# → 需要网络、需要 API key、可能花钱、依赖外部服务

用 Mock 后：

def test_chat_with_mock(mocker):
    client = LLMClient()

    # 1. 创建一个假的 API 响应
    mock_response = mocker.Mock()
    mock_response.choices = [mocker.Mock()]
    mock_response.choices[0].message.content = "mocked reply"

    # 2. 把真正的 API 调用替换成假的
    mocker.patch.object(
        client._client.chat.completions,
        "create",
        return_value=mock_response
    )

    # 3. 现在调用 chat() 不会真的调 API，而是返回 mock_response
    reply = client.chat("你好")
    assert reply == "mocked reply"  # ✅ 测试 chat() 的逻辑
    assert client.history_length == 3  # ✅ 测试历史管理

Mock 的好处：

  • 不需要网络、API key、费用
  • 速度快（毫秒级）
  • 稳定（不依赖外部服务状态）
  • 可以模拟各种极端情况（超时、报错、空回复等）
25 mocker.patch.object() 是做什么的？ ▼

pytest

mocker.patch.object() 把真实对象的某个方法临时替换成假的。

mocker.patch.object(
    client._client.chat.completions,  # 目标对象
    "create",                         # 要替换的方法名
    return_value=mock_response        # 替换后的返回值
)

# 替换前：client._client.chat.completions.create(...) → 调真实 API
# 替换后：client._client.chat.completions.create(...) → 返回 mock_response

其他变体：

# 替换模块里的一个函数
mocker.patch("os.getenv", return_value="fake_key")

# 替换整个类
mocker.patch("openai.OpenAI")

# 让 mock 在被调用时执行特定函数
mocker.patch.object(obj, "method", side_effect=my_func)

重要：替换只在当前测试函数内生效，测试结束后自动恢复。不会影响其他测试。

26 Mock 如何模拟 API 响应？response.choices[0].message.content 这个结构怎么来的？ ▼

pytest

这个结构来自 OpenAI API 的响应格式：

# 真实的 API 响应结构（简化版）
{
    "choices": [
        {
            "message": {
                "content": "这是AI的回复",
                "role": "assistant"
            }
        }
    ]
}

# 在 Python SDK 中通过对象属性访问
response.choices[0].message.content

Mock 模拟的写法：

# 创建最外层的 Mock 对象
mock_response = mocker.Mock()

# 模拟 choices[0]：choices 是一个列表
mock_response.choices = [mocker.Mock()]

# 模拟 choices[0].message.content
mock_response.choices[0].message.content = "mocked reply"

# 现在 mock_response 的结构和真实响应一样
# chat() 方法中 response.choices[0].message.content 就能拿到 "mocked reply"

为什么能这样模拟？因为 Mock 对象有个特点：访问任何属性都不会报错，而是自动创建子 Mock。

mock = mocker.Mock()
mock.a.b.c.d  # ✅ 不会报错，自动创建链式 Mock
mock.a.b.c.d = "hello"  # ⭐ 再赋值即可
