1 同步（Synchronous）和异步（Asynchronous）的本质区别是什么？ ▼

异步编程

核心区别：阻塞 vs 不阻塞

同步：做完一件事再做下一件事。如果某一步在等待（比如等 API 响应），整个程序就停在那里干等。

# 同步：一次一个，顺序执行
print("开始")
response = requests.get("https://api.example.com")  # 卡在这里等 2 秒
print("结束")  # 2 秒后才执行

异步：在等待的时候去做别的事。就像一边烧水一边切菜——不浪费等待时间。

# 异步：遇到等待就切换任务
async def main():
    print("开始")
    response = await client.get("https://api.example.com")  # 不阻塞，去干别的
    print("结束")  # 等数据回来再继续

asyncio.run(main())

类比生活：

  • 同步：去银行排队，在窗口前等到办完才能走
  • 异步：取号后去旁边坐着玩手机，叫到号才过去

对于 LLM 调用：API 请求大部分时间花在网络传输上（IO 等待），异步可以让 CPU 在等待期间处理其他请求，从而大幅提升并发能力。

2 事件循环（Event Loop）是什么？它怎么工作？ ▼

异步编程

事件循环是 Python 异步编程的核心引擎。它是一个无限循环，不断做两件事：

  1. 检查有没有"可以继续执行"的协程
  2. 如果有，就让它执行一段；如果没有，就等一会儿再检查
# 事件循环的简化理解
tasks = [task1, task2, task3]  # 待执行的协程列表

while True:  # 事件循环
    for task in tasks:
        if task.is_ready():      # 这个任务可以继续了吗？
            task.run_one_step()  # 让它执行一步（直到下一个 await）
        # 如果不可执行（还在等 IO），就跳过，检查下一个

关键理解：

  • 事件循环运行在单线程上
  • 它不靠多线程/多进程，而是靠"调度"——在任务之间快速切换
  • 切换发生的时机：遇到 await 时（即 IO 等待时）
  • 因为切换成本极低（比线程切换低得多），所以能高效处理大量并发

启动事件循环：asyncio.run(main()) ← 一切的入口。

3 async def 和 await 分别是什么？怎么用？ ▼

异步编程

async def：定义一个协程函数（coroutine function）。调用它不会立即执行，而是返回一个协程对象。

async def fetch_data():
    # 这是一个协程函数
    return "数据"

coro = fetch_data()  # ⚠️ 还没有执行！只是创建了一个协程对象
# 必须 await 或用 asyncio.run 才能执行

await：等待一个协程完成，不阻塞线程。遇到 await 时，事件循环可以切换到其他任务。

async def main():
    # await 会暂停当前协程，但不阻塞整个程序
    result = await fetch_data()  # 等 fetch_data 完成
    print(result)  # → "数据"

asyncio.run(main())

await 的规则：

  • await 只能用在 async def 内部
  • await 后面只能跟"可等待对象"（coroutine、Task、Future）
  • 普通函数里不能写 await → SyntaxError

和 Swift 的对比：完全一样——Swift 的 async func 对应 async def，try await 对应 await。

4 asyncio.run() 是做什么的？为什么需要它？ ▼

异步编程

asyncio.run() 是启动事件循环的入口函数。

import asyncio

async def hello():
    return "Hello, Async!"

# ❌ 不能直接调用协程函数
result = hello()  # 返回一个协程对象，不是字符串

# ✅ 必须用 asyncio.run() 来运行
result = asyncio.run(hello())  # → "Hello, Async!"

它做了什么事？

  1. 创建新的事件循环
  2. 把传入的协程加入事件循环并执行
  3. 等待协程完成，返回结果
  4. 关闭事件循环

重要规则：

  • 一个线程里只能有一个事件循环
  • asyncio.run() 不能被嵌套调用
  • 通常只写一次 asyncio.run(main()) 作为程序的入口
# ✅ 正确写法
async def main():
    ...
asyncio.run(main())

# ❌ 错误写法
asyncio.run(asyncio.run(other()))  # 嵌套调用会报错
5 asyncio.gather() 是什么？它怎么让多个 LLM 请求并发执行？ ▼

异步编程

asyncio.gather() 并发执行多个协程，等待所有完成，返回结果列表。

import asyncio

async def fetch_data(id: int) -> str:
    await asyncio.sleep(1)  # 模拟 API 调用
    return f"数据 {id}"

async def main():
    # 串行执行：3 个请求要 3 秒
    # r1 = await fetch_data(1)
    # r2 = await fetch_data(2)
    # r3 = await fetch_data(3)

    # 并发执行：3 个请求只要 1 秒
    results = await asyncio.gather(
        fetch_data(1),
        fetch_data(2),
        fetch_data(3),
    )
    print(results)  # → ['数据 1', '数据 2', '数据 3']

asyncio.run(main())

在 LLM 场景中的威力：

# 同步：10 个请求 → 10 * 平均响应时间（假设 3 秒）= 30 秒
responses = [sync_call(p) for p in prompts]

# 异步：10 个请求 → 最快 ≈ 3 秒（最慢的那个响应时间）
results = await asyncio.gather(*[async_call(p) for p in prompts])

gather 的细节：

  • *[coro1, coro2, ...] 解包列表为多个参数
  • 如果某个协程抛出异常，gather 默认会立刻抛出（不会等其他的）
  • 可以用 return_exceptions=True 让 gather 返回异常对象而非抛出
6 asyncio.create_task() 和 gather() 有什么区别？ ▼

异步编程

create_task()：把协程包装成 Task，立即开始后台执行，不等待结果。

async def background_work():
    await asyncio.sleep(5)
    print("后台任务完成")

async def main():
    # 创建后台任务，立刻开始执行
    task = asyncio.create_task(background_work())
    print("主任务继续...")
    await asyncio.sleep(1)
    print("主任务结束")
    # 如果主任务结束前想等后台任务完成
    await task

asyncio.run(main())
 gather()create_task()
用途并发执行一批协程，等全部完成把协程丢到后台运行，不阻塞当前
返回值所有结果组成的列表Task 对象
典型场景同时发 N 个 API 请求后台日志、监控、定时任务
等待方式await gather(...)await task

实际关系：gather() 内部也会创建 Task，可以理解为 gather 是 "create_task + 统一 await" 的快捷方式。

7 async with 和 async for 是什么？ ▼

异步编程

async with：异步上下文管理器（类似同步的 with，但进入和退出都是异步的）。

# 同步 with
with open("file.txt") as f:
    data = f.read()

# 异步 async with（用于异步资源，如 HTTP 客户端、数据库连接）
async with httpx.AsyncClient() as client:
    resp = await client.get("https://example.com")

# AsyncOpenAI 也是用 async with
client = AsyncOpenAI(...)  # 或者用 async with

async for：异步迭代器（每次迭代都可能需要等待）。

# 异步流式读取 LLM 回复
async for chunk in stream:
    print(chunk, end="", flush=True)

# 等价于同步的：
# for chunk in stream:
#     print(chunk, end="")

# 区别：async for 的每一步内部可能 await 等待数据到达

记忆方法：在同步版本前加 async 就是异步版本——with → async with，for → async for。

8 为什么 LLM 调用必须用异步而非多线程？ ▼

异步编程

一句话答案：LLM API 调用是 IO 密集型任务（绝大多数时间花在网络传输上），不是 CPU 密集型。

 多线程异步 async/await
并发单位操作系统线程协程（用户态）
切换成本高（内核态切换，~1μs）极低（用户态切换，~0.01μs）
内存开销每个线程 ~8MB 栈每个协程 ~几 KB
适合场景CPU 密集型（视频渲染、计算）IO 密集型（网络请求、数据库）
Python 的 GIL受 GIL 限制，不能真正并行 CPU 任务不受 GIL 影响

具体到 LLM 调用：

  • 一次 API 调用：99% 时间在等网络，1% 时间在处理数据
  • 同步 10 次调用：10 * 3 秒 = 30 秒
  • 多线程 10 次调用：接近 3 秒（但线程开销大，创建 1000 个线程会压垮系统）
  • 异步 10 次调用：接近 3 秒（创建 10000 个协程也轻松）

所以异步是 IO 密集型任务的最优解——低成本、高并发、不阻塞。

11 Python 类型标注（Type Hints）是什么？有什么用？ ▼

类型标注

类型标注是 Python 3.5+ 引入的可选语法，用来标记变量、参数、返回值的类型。

# 不加类型标注
def add(x, y):
    return x + y

# 加类型标注
def add(x: int, y: int) -> int:
    return x + y

有什么用：

  1. IDE 智能补全：Trae 知道变量类型后，会给出精确的代码提示
  2. 静态检查：用 mypy 可以在运行前发现类型错误
  3. 代码即文档：别⼈一看函数签名就知道怎么用
  4. 减少 bug：很多隐式类型错误在写代码时就被发现

重要：类型标注是可选的，Python 运行时不强制检查类型。它主要用于开发工具和静态检查器。

12 Python 3.9+ 的类型标注怎么写？list[int] 和 List[int] 有什么区别？ ▼

类型标注

Python 3.9+ 可以直接用内置类型作为泛型，不需要从 typing 导入：

# Python 3.9+（推荐写法）
nums: list[int] = [1, 2, 3]
mapping: dict[str, int] = {"a": 1}
pair: tuple[str, int] = ("a", 1)
items: set[str] = {"a", "b"}

# Python 3.8 及以前（需要从 typing 导入）
from typing import List, Dict, Tuple, Set
nums: List[int] = [1, 2, 3]
mapping: Dict[str, int] = {"a": 1}
pair: Tuple[str, int] = ("a", 1)
items: Set[str] = {"a", "b"}

区别：

  • list[int]（小写）— Python 3.9+ 内置，不需要 import
  • List[int]（大写）— 旧式写法，需要 from typing import List
  • 功能完全一样，推荐用新的小写写法
13 Optional、Union、Literal 分别是什么？怎么用？ ▼

类型标注

Optional[X]：表示值可以是 X 类型或 None（等价于 X | None）。

from typing import Optional

name: Optional[str] = None  # 可能是 str 或 None
# Python 3.10+ 也可写：name: str | None = None

def find_user(id: int) -> Optional[dict]:
    if id == 1:
        return {"name": "小明"}
    return None  # 可能返回 None

Union[X, Y]：表示值可以是 X 或 Y 类型（等价于 X | Y）。

from typing import Union

value: Union[int, str] = 1    # 可以是 int 或 str
value = "hello"               # 也是合法的

# Python 3.10+ 也可写：value: int | str = 1

Literal["a", "b"]：表示值只能是指定的字面量值之一。AI 库中大量使用！

from typing import Literal

role: Literal["user", "assistant", "system"] = "user"
role = "student"  # ❌ mypy 会报错（必须是 "user"/"assistant"/"system" 之一）

# 在 LLMClient 中的实际应用
Role = Literal["user", "assistant", "system"]

@dataclass
class ChatMessage:
    role: Role       # 只能是这三个值之一
    content: str

联合类型的新语法（3.10+）：

# 旧
x: Optional[int] = None
y: Union[int, str] = 1

# 新（更简洁）
x: int | None = None
y: int | str = 1
14 Callable 和 TypedDict 是什么？怎么用？ ▼

类型标注

Callable[[参数类型], 返回类型]：标注函数的类型。

from typing import Callable

# 定义一个"接收两个 int 返回 bool"的函数类型
handler: Callable[[int, int], bool] = lambda x, y: x > y

# 在函数参数中使用
def apply(func: Callable[[int], int], value: int) -> int:
    return func(value)

apply(lambda x: x * 2, 5)  # → 10

TypedDict：给字典定义结构化的类型（像 Swift 的 struct 但用字典表示）。

from typing import TypedDict

class Message(TypedDict):
    role: str
    content: str

# 使用 TypedDict
msg: Message = {"role": "user", "content": "你好"}
msg["role"]         # 可访问
msg["invalid"]      # ❌ mypy 报错：extra key 'invalid'

# 相比普通 dict 的好处：IDE 能补全 key，mypy 能检查 key 是否存在
normal_dict: dict = {"role": "user"}
normal_dict["xxx"]  # ✅ 不报错（运行时才能发现）

Callable 和 Swift 对比：

# Swift: typealias Handler = (Int, Int) -> Bool
# Python: handler: Callable[[int, int], bool]
15 mypy 是什么？怎么用它对 Python 代码做静态类型检查？ ▼

类型标注

mypy 是 Python 的静态类型检查器，相当于 Swift 的编译器类型检查。

安装：

pip install mypy

使用：

# 检查单个文件
mypy my_script.py

# 检查整个目录
mypy my_package/

# 忽略 import 错误
mypy --ignore-missing-imports my_script.py

# 严格模式
mypy --strict my_script.py

实际效果：

# file: demo.py
def greet(name: str) -> str:
    return name + 1  # ❌ str + int，类型错误

age: int = "twenty"  # ❌ str 赋值给 int，类型错误

16 Any 是什么？什么时候该用它？ ▼

类型标注

Any 表示"任意类型"。当变量标注为 Any 时，mypy 不会对它做任何类型检查。

from typing import Any

value: Any = 1
value = "hello"       # ✅ 任何赋值都可以
value.some_method()   # ✅ mypy 不会报错（运行时可能崩溃）

# ❌ 不推荐这样用，等于放弃了类型检查的好处
# ✅ 只在以下情况用：
# 1. 和动态类型库交互（如某些 JSON 解析）
# 2. 迁移旧代码时临时过渡
# 3. 确实无法确定类型的情况

最佳实践：尽量少用 Any。能用 Union 或 Optional 代替就用。每多一个 Any，就多一份运行时崩溃的风险。

17 Python 装饰器的本质是什么？它和闭包是什么关系？ ▼

装饰器

装饰器的本质：就是一个接收函数、返回新函数的高阶函数。

# 装饰器 = 接收函数，返回新函数
def decorator(func):
    def wrapper(*args, **kwargs):
        # 在调用原函数前/后做点额外的事
        result = func(*args, **kwargs)
        return result
    return wrapper

# @ 语法糖
@decorator
def my_func():
    pass

# 等价于
my_func = decorator(my_func)

和闭包的关系：

  • 闭包：内部函数引用外部函数的变量——一种编程机制
  • 装饰器：用闭包包装另一个函数的模式——闭包的典型应用
  • 装饰器中的 wrapper 函数就是一个闭包（它引用了外部的 func 变量）
18 怎么徒手写一个 @timer 装饰器来测函数执行时间？ ▼

装饰器

基础版 @timer：

import time

def timer(func):
    """计算函数执行时间的装饰器"""
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        elapsed = time.time() - start
        print(f"{func.__name__} 耗时: {elapsed:.4f}s")
        return result
    return wrapper

# 使用
@timer
def slow_function():
    time.sleep(1)
    return "完成"

slow_function()  # 输出: slow_function 耗时: 1.0032s

进阶版：支持异步函数的 @timer：

import time
import asyncio
import functools

def async_timer(func):
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        start = time.time()
        result = await func(*args, **kwargs)
        elapsed = time.time() - start
        print(f"⏱ {func.__name__} 耗时 {elapsed:.2f}s")
        return result
    return wrapper

@async_timer
async def fetch_data():
    await asyncio.sleep(1)
    return "数据"

asyncio.run(fetch_data())

注意：异步装饰器的 wrapper 必须是 async def，内部用 await 调用原函数。

19 @functools.wraps 是什么？为什么写装饰器必须加它？ ▼

装饰器

@functools.wraps 把原函数的元信息（__name__、__doc__ 等）复制到 wrapper 函数上。

# ❌ 不加 @wraps 的后果
def timer(func):
    def wrapper(*args, **kwargs):
        """我是 wrapper 的文档"""
        ...
        return func(*args, **kwargs)
    return wrapper

@timer
def hello():
    """我是 hello 的文档"""
    print("Hello")

print(hello.__name__)  # → "wrapper"（不是 "hello"！）
print(hello.__doc__)   # → "我是 wrapper 的文档"（原文档丢失了！）
# ✅ 加上 @wraps
import functools

def timer(func):
    @functools.wraps(func)  # ← 复制 func 的元信息到 wrapper
    def wrapper(*args, **kwargs):
        ...
        return func(*args, **kwargs)
    return wrapper

@timer
def hello():
    """我是 hello 的文档"""
    print("Hello")

print(hello.__name__)  # → "hello" ✅
print(hello.__doc__)   # → "我是 hello 的文档" ✅

为什么重要：调试时 __name__ 保留原函数名，否则所有被装饰的函数都显示为 wrapper，无法区分。

20 带参数的装饰器怎么写？比如 @retry(times=3)？ ▼

装饰器

带参数的装饰器需要三层嵌套：外层接收参数，中层接收函数，内层是 wrapper。

def retry(times: int = 3, delay: float = 0):
    """失败重试装饰器"""
    def decorator(func):           # 中层：接收函数
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for i in range(times):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if i == times - 1:
                        raise  # 最后一次失败，抛出异常
                    print(f"重试 {i+1}/{times}: {e}")
                    if delay:
                        time.sleep(delay)
            return None
        return wrapper
    return decorator               # 返回中层

# 使用
@retry(times=3, delay=1)
def unstable_api():
    import random
    if random.random() < 0.7:
        raise ConnectionError("网络错误")
    return "成功"

# 也可以不传参使用默认值
@retry()
def another_func():
    ...

# 工作原理：
# @retry(times=3) 等价于 retry(times=3)  → 返回 decorator
# decorator 再接收函数 → 返回 wrapper
21 Python 有哪些常见的内置装饰器？分别有什么用？ ▼

装饰器

装饰器作用
@property把方法变成属性访问（不加括号）
@staticmethod静态方法（不需要 self）
@classmethod类方法（接收 cls 而不是 self）

@cached_property 示例：

from functools import cached_property

class DataProcessor:
    def __init__(self, data):
        self.data = data

    @cached_property
    def processed(self):
        """计算结果会被缓存，第二次访问直接返回缓存"""
        print("处理大数据...")
        time.sleep(2)  # 模拟耗时
        return [x * 2 for x in self.data]

dp = DataProcessor([1, 2, 3])
print(dp.processed)  # 慢（实际计算）
print(dp.processed)  # 快（返回缓存值）

22 类装饰器（class decorator）和函数装饰器有什么区别？ ▼

装饰器

类装饰器是应用于类的装饰器，接收类、返回修改后的类或新类。

# 函数装饰器：接收函数，返回函数
def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        ...
    return wrapper

# 类装饰器：接收类，返回类
def add_repr(cls):
    """给类自动添加 __repr__ 方法"""
    def __repr__(self):
        items = ", ".join(f"{k}={v}" for k, v in self.__dict__.items())
        return f"{cls.__name__}({items})"
    cls.__repr__ = __repr__
    return cls

@add_repr
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(3, 4)
print(p)  # → Point(x=3, y=4)  （自动生成了 __repr__）

@dataclass 就是一个类装饰器——它接收一个类，自动添加 __init__、__repr__、__eq__ 等方法，返回增强后的类。

23 装饰器的执行顺序是什么？多个装饰器叠加时怎么工作？ ▼

装饰器

多个装饰器叠加时，执行顺序是"从下往上"装饰，"从上往下"执行。

@decorator_a
@decorator_b
@decorator_c
def my_func():
    pass

# 实际等价于：
my_func = decorator_a(decorator_b(decorator_c(my_func)))
#          从下往上：先装饰 c → 再装饰 b → 最后装饰 a

# 调用时执行顺序：
# 1. decorator_a 的 wrapper 开始执行
# 2. → decorator_b 的 wrapper 开始执行
# 3. → decorator_c 的 wrapper 开始执行
# 4. → 真正的 my_func 执行
# 5. → decorator_c 的 wrapper 结束
# 6. → decorator_b 的 wrapper 结束
# 7. → decorator_a 的 wrapper 结束
# 像洋葱一样，一层包一层

实际例子：

def bold(func):
    @functools.wraps(func)
    def wrapper():
        return f"{func()}"
    return wrapper

def italic(func):
    @functools.wraps(func)
    def wrapper():
        return f"{func()}"
    return wrapper

@bold
@italic
def hello():
    return "Hello"

print(hello())  # → Hello

# 装饰顺序：italic 先装饰 → bold 再装饰
# 执行顺序：bold 先 → italic → hello → italic → bold
24 同步 vs 异步并发对比实验怎么设计？关键代码怎么写？ ▼

实战

实验思路：用同样的 10 个问题，分别用同步和异步方式调用 LLM，对比总耗时。

同步版核心代码：

def sync_call(prompt: str) -> str:
    client = OpenAI(api_key=API_KEY, base_url=API_BASE)
    resp = client.chat.completions.create(
        model=MODEL,
        messages=[{"role": "user", "content": prompt}],
    )
    return resp.choices[0].message.content

def run_sync():
    start = time.time()
    results = [sync_call(p) for p in PROMPTS]     # 逐个串行
    elapsed = time.time() - start
    print(f"同步耗时: {elapsed:.2f}s")
    return elapsed

异步版核心代码：

async def async_call(client: AsyncOpenAI, prompt: str) -> str:
    resp = await client.chat.completions.create(
        model=MODEL,
        messages=[{"role": "user", "content": prompt}],
    )
    return resp.choices[0].message.content

async def run_async():
    client = AsyncOpenAI(api_key=API_KEY, base_url=API_BASE)
    start = time.time()
    results = await asyncio.gather(    # 并发执行！
        *[async_call(client, p) for p in PROMPTS]
    )
    elapsed = time.time() - start
    print(f"异步耗时: {elapsed:.2f}s")
    return elapsed

# 主程序
if __name__ == "__main__":
    t_sync = run_sync()
    t_async = asyncio.run(run_async())
    print(f"异步比同步快 {t_sync / t_async:.1f} 倍")

预期结果：

  • 同步：10 个请求逐个执行，总耗时 ≈ 10 * 单个响应时间（如 30 秒）
  • 异步：10 个请求并发执行，总耗时 ≈ 最慢的那个响应时间（如 3 秒）
  • 加速比通常在 5-10 倍

关键差异点：

  • 同步用 OpenAI(...) 客户端 + 普通 for 循环
  • 异步用 AsyncOpenAI(...) 客户端 + asyncio.gather()
  • 异步要用 asyncio.run() 启动事件循环

27 AsyncLLMClient 的 @async_timer 装饰器是怎么工作的？ ▼

实战

@async_timer 是一个异步函数专用的计时装饰器：

def async_timer(func):
    """异步函数计时装饰器"""
    @functools.wraps(func)             # 保留原函数的 __name__ 等元信息
    async def wrapper(*args, **kwargs):
        start = time.time()            # 开始计时
        result = await func(*args, **kwargs)  # await 原异步函数
        elapsed = time.time() - start  # 结束计时
        print(f"⏱ {func.__name__} 耗时 {elapsed:.2f}s")
        return result                  # 返回原函数的结果
    return wrapper

# 使用
class AsyncLLMClient:
    @async_timer
    async def chat(self, user_input: str) -> str:
        ...  # 调用时自动计时并打印耗时

和同步 @timer 的区别：

 @timer（同步）@async_timer（异步）
wrapper 类型def wrapper(...)async def wrapper(...)
调用原函数func(*args)await func(*args)
适用范围普通函数async def 定义的协程函数

@functools.wraps 的作用：如果不加，被装饰的函数的 __name__ 会变成 wrapper，而不是原本的 chat，影响调试和日志。
